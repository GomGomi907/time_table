# AI 스케줄 오퍼레이터 리부트 사용자 페르소나 정의서

## 1. 문서 목적
- 이 문서는 제품의 1차 사용자상을 선명하게 고정하기 위한 기준 문서다.
- 기능 우선순위, 화면 정보 위계, AI 개입 톤, 입력 플로우, 온보딩 구조는 이 문서의 페르소나를 기준으로 결정한다.
- 이번 리부트의 핵심 사용자는 “일정 기록자”가 아니라 “자기 시간을 직접 운영하는 솔로 오퍼레이터”다.

## 2. 1차 페르소나

### 2.1 기본 프로필
| 항목 | 내용 |
| --- | --- |
| 이름 | 박준호 |
| 나이 | 29 |
| 직업 | 주간 근무를 하는 직장인, 퇴근 후 사이드 프로젝트와 공부를 병행 |
| 거주/생활 | 통근 시간이 있고, 평일과 주말의 시간 밀도가 크게 다름 |

| 성향 | 목표지향적, 자기 기준이 높고, 계획이 어그러지는 것을 싫어함 |
| 결정 방식 | 완벽주의보다 운영 가능성을 중시. “지킬 수 있는 계획”을 선호 |

### 2.2 한 줄 정의
- 준호는 목표가 없는 사람이 아니라, 목표는 많은데 현실 제약과 피로 때문에 계획 운영이 자주 무너지는 사람이다.

### 2.3 하루의 구조
| 시간대 | 현실 |
| --- | --- |
| 오전 | 출근 준비와 이동으로 고정 제약이 강함 |
| 낮 | 회사 일정, 회의, 돌발 요청으로 자율 시간 확보가 어려움 |
| 저녁 | 퇴근 후 가장 중요한 개인 시간대지만 피로와 일정 밀림이 겹침 |
| 심야 | 공부나 사이드 프로젝트를 밀어넣기 쉬우나, 과부하와 수면 손실 위험이 큼 |
| 주말 | 회복, 약속, 알바 또는 집안일이 섞이며 계획 재정비 시간이 필요 |

### 2.4 현재 도구 사용 방식
- Google Calendar에는 회의, 약속, 이동 일정이 들어 있다.
- 노트 앱이나 메신저 저장 메시지에 “나중에 해야 할 일”이 흩어져 있다.
- 투두 앱에는 해야 할 일이 많지만, 실제 시간표까지 연결되지는 않는다.
- 계획을 세울 때는 캘린더, 메모, 머릿속 우선순위를 계속 왔다 갔다 한다.

## 3. 심층 행동 패턴

### 3.1 행동 특징
- 일정을 세울 때보다, 일정이 무너진 뒤 다시 짜는 데 더 큰 스트레스를 느낀다.
- 남은 시간을 보면 “뭔가 더 넣을 수 있다”고 생각해 계획을 과적재하는 경향이 있다.
- 중요한 목표는 알고 있지만, 당장 눈앞의 급한 일에 쉽게 먹힌다.
- 실행 도중 다른 화면이나 다른 앱으로 이동하면 집중이 많이 깨진다.
- AI를 좋아하지만, 마법처럼 다 해주는 것보다 “왜 이렇게 제안했는지 설명 가능한 보조”를 더 신뢰한다.

### 3.2 반복되는 실패 패턴
- 외부 일정이 하나 추가되면 전체 계획이 연쇄적으로 밀린다.
- 저녁 핵심 블록이 늦어지면 뒤 일정 중 무엇을 포기해야 할지 즉시 판단하지 못한다.
- 인박스 작업이 쌓여도 오늘 할 일로 끌어오는 기준이 불분명하다.
- 목표는 세워도 오늘 시간표와 연결되지 않아 성취감이 약하다.
- 캘린더와 투두가 분리되어 있어 “오늘 현실적으로 가능한 계획”을 한 번에 보기 어렵다.

### 3.3 감정 곡선
| 상황 | 감정 |
| --- | --- |
| 주간 계획을 세울 때 | 기대감, 통제감 |
| 업무나 약속이 끼어들 때 | 초조함, 분산 |
| 밤에 계획이 밀린 것을 볼 때 | 죄책감, 포기 충동 |
| 다시 정리할 때 | 귀찮음, 인지 피로 |
| 명확한 다음 행동을 받을 때 | 안도감, 복귀 가능성 |

## 4. 목표와 동기

### 4.1 상위 목표
- 장기적으로 성장하고 있다는 감각을 잃지 않는 것
- 본업, 자기계발, 사이드 프로젝트를 동시에 굴릴 수 있는 운영 체계를 갖는 것
- 의욕이 아니라 시스템으로 하루를 굴리는 것

### 4.2 제품에 기대하는 결과
- 지금 해야 할 일이 명확하게 보여야 한다.
- 계획이 틀어졌을 때 즉시 대안을 받아야 한다.
- 오늘의 시간이 어떤 목표를 위해 쓰이는지 바로 이해되어야 한다.
- 외부 일정과 개인 계획이 분리되지 않고 함께 보여야 한다.

### 4.3 성공 판단 기준
- 하루 시작 전에 오늘 계획을 5분 안에 확정할 수 있다.
- 실행 중 계획이 틀어져도 1분 안에 다음 액션을 결정할 수 있다.
- 주간 목표가 실제 시간 사용과 연결되어 있다고 느낀다.
- 하루를 마친 뒤 “많이 적어두었다”가 아니라 “실제로 굴러갔다”고 느낀다.

### 5.1 Functional Jobs
- 내 캘린더와 루틴을 기준으로 실제 가능한 시간을 계산하고 싶다.
- 목표와 작업을 오늘 시간표로 자동 연결하고 싶다.
- 일정이 꼬였을 때 연장, 이동, 삭제 후보를 빠르게 알고 싶다.
- 인박스에 쌓인 일들을 오늘 안건과 나중 안건으로 분리하고 싶다.

### 5.2 Emotional Jobs
- 하루가 밀려도 완전히 망한 느낌 대신 “아직 복구 가능하다”는 감각을 받고 싶다.
- AI가 나를 통제하는 느낌이 아니라, 내 판단을 빠르게 도와준다는 느낌을 받고 싶다.
- 복잡한 계획 대신 현재 실행 가능한 한 걸음에 몰입하고 싶다.

### 5.3 Social Jobs
- 스스로를 게으른 사람이 아니라 운영 체계를 가진 사람으로 느끼고 싶다.
- “할 일 많다”가 아니라 “우선순위를 관리하고 있다”는 자기 인식을 갖고 싶다.

## 6. 결정 기준과 신뢰 조건

### 6.1 신뢰를 높이는 요소
- AI가 제안 이유를 짧고 명확하게 설명한다.
- 어떤 일정이 왜 보호되고 왜 밀렸는지 보인다.
- 사용자가 승인하기 전에는 중요한 일정이 자동으로 뒤집히지 않는다.
- 캘린더 연동 상태와 누락 위험이 솔직하게 드러난다.

### 6.2 신뢰를 깨는 요소
- AI가 몰래 시간을 바꾸는 것
- 계획을 화려하게 보여주지만 실제로는 실행을 도와주지 못하는 것
- 정보가 너무 많아서 지금 할 일이 묻히는 것
- 실패했을 때 원래 상태나 이유를 알 수 없는 것

## 7. 대표 시나리오

### 7.1 시나리오 A: 평일 저녁 복구
- 오후 7시 퇴근 후 원래는 8시부터 공부 블록을 시작할 계획이었다.
- 갑자기 회의가 길어져 귀가가 늦어졌고, 저녁 루틴도 함께 밀렸다.
- 준호는 무엇을 포기하고 무엇을 살릴지 고민하다 시간을 더 날린다.
- 이때 제품은 현재 상황을 바탕으로 `AgentIntervention`을 띄우고, “공부 블록 20분 축소 + 영어 회화는 내일 오전 후보로 이동” 같은 복구안을 제안해야 한다.

### 7.2 시나리오 B: 주간 목표와 오늘 실행 연결
- 준호는 “MVP 기획 정리”라는 주간 목표는 세웠지만 오늘 무엇을 해야 하는지는 모호하다.
- 제품은 목표를 오늘 후보 작업으로 분해하고, 가용 시간에 맞게 `ScheduledBlock`으로 연결해야 한다.
- 워크스페이스에서는 현재 블록이 어느 목표에 연결되는지 항상 보여줘야 한다.

### 7.3 시나리오 C: 연동 실패 fallback
- Google Calendar 토큰이 만료되어 오늘 회의가 누락될 수 있다.
- 제품은 연동이 건강하지 않음을 명확히 알리고, 즉시 수동 `FixedEvent` 추가 경로를 열어야 한다.
- 준호는 연동 실패 자체보다, 그 때문에 오늘 계획 전체를 다시 손으로 짜야 하는 상황을 더 싫어한다.

## 8. 2차 페르소나와 경계

### 8.1 2차 페르소나: 계획형 학생
- 학업, 과제, 시험 준비를 중심으로 돌아간다.
- 주간 목표 분해와 포커스 실행 니즈는 강하지만 외부 회의성 `FixedEvent`는 상대적으로 적다.
- 이 사용자는 제품에 잘 맞지만, 1차 기준을 학생 중심으로 재설계하지는 않는다.

### 8.2 비대상 사용자
- 팀 단위 협업과 공유 일정이 핵심인 사용자
- 예쁜 위젯 커스터마이징 자체가 제품 사용 이유인 사용자
- 장기 아카이빙, 회고 리포트, 성과 분석이 핵심인 사용자
- AI에게 완전 자동 운영을 맡기고 세부 결정을 거의 하지 않으려는 사용자

## 9. 제품 설계 함의
- 메인 화면은 기록보다 실행 중심이어야 한다.
- 포커스 보드는 반드시 현재 작업, 목표 연결, 다음 선택지를 함께 보여야 한다.
- AI 개입은 빈번해도 되지만 장황하면 안 된다. 짧고 결정 가능한 형태여야 한다.
- 온보딩은 캘린더 연결, 루틴 입력, 목표 설정, 개입 정책을 빠르게 끝내야 한다.
- 목표 페이지는 동기부여용 전시장이 아니라 오늘 후보 작업을 만드는 운영 도구여야 한다.
- 설정 페이지는 복잡한 옵션 창이 아니라 신뢰와 제어감을 복구하는 화면이어야 한다.
