# Spring Boot Project Midterm Presentation 설계 브리프

## 1. 문서 목적
- 이 문서는 `Project Midterm Presentation`을 다시 만들기 위한 기준 문서다.
- 현재 덱의 문제는 단순히 예쁘지 않은 것이 아니라, 발표 자료로서의 정보 위계와 시선 흐름이 무너진 데 있다.
- 다음 리디자인 단계에서는 이 문서의 `내용 구성`, `디자인 철학`, `색상 팔레트`, `레이아웃 규칙`을 그대로 따른다.

## 2. 현재 덱의 핵심 문제
- 한 슬라이드 안에 텍스트 블록이 너무 많아 시선이 분산된다.
- 제목, 부제, 카드, 캡션, 목업이 동시에 강조되어 무엇이 핵심인지 바로 읽히지 않는다.
- 어두운 배경과 낮은 대비 텍스트가 많아 시인성이 떨어진다.
- 목업 슬라이드는 좌측 설명 카드와 우측 목업이 모두 크고 강해서 F-패턴이 깨진다.
- 발표용 슬라이드인데 기획 문서와 대시보드가 한 장에 섞인 느낌이라, “한 슬라이드 = 한 메시지” 원칙이 없다.

## 3. 발표 목표

### 3.1 발표의 역할
- 이 자료는 완성본 시연 자료가 아니라, 프로젝트를 제안하고 소개하는 `중간발표용 기획 자료`다.
- 따라서 “무엇을 왜 만들고, 어떤 구조로 구현하며, 지금 어디까지 왔는가”를 명확하게 전달해야 한다.

### 3.2 청중
- 수업 담당 교수
- 같은 수업을 듣는 학생
- 구현 세부 코드보다 문제 정의, 설계 타당성, 구현 가능성, 중간 진행도를 보고 싶은 사람들

### 3.3 발표 한 줄 메시지
- `AI가 오늘 일정을 다시 운영해 주는 Spring Boot 기반 개인 스케줄 보조 시스템을 제안한다.`

## 4. 내용 구성 원칙

### 4.1 한 슬라이드 한 메시지
- 슬라이드 한 장은 하나의 질문에만 답해야 한다.
- 한 장 안에서 `문제 정의 + 사용자 + 기능 설명`을 동시에 하지 않는다.

### 4.2 텍스트 밀도 제한
- 본문은 슬라이드당 최대 3개 포인트만 허용한다.
- 한 포인트는 한 줄 또는 두 줄을 넘기지 않는다.
- 긴 문단 대신 `헤드라인 + 짧은 설명` 구조를 쓴다.

### 4.3 시각자료 우선
- 모든 슬라이드는 텍스트만 두지 않는다.
- 다만 시각자료는 설명을 보조해야지, 읽을 텍스트를 더 늘리는 장식이 되면 안 된다.

### 4.4 중간발표에 맞는 범위 통제
- 마켓플레이스, 팀 협업, 확장 SaaS 서사는 이번 덱에서 제외한다.
- 이번 발표 범위는 `문제 정의`, `핵심 사용자`, `핵심 솔루션`, `핵심 기능`, `UI/UX 방향`, `Spring Boot 구현 계획`, `중간 진행 상황`까지다.

## 5. 권장 슬라이드 구성

### Slide 1. 제목 / 프로젝트 한 줄 소개
**목적**
- 발표 시작 10초 안에 “무슨 프로젝트인지”를 이해시키기

**핵심 메시지**
- AI가 일정 변경 상황에서 다음 행동을 조율 및 제안하는 개인용 스케줄 보조 시스템

**들어갈 내용**
- 프로젝트명
- 한 줄 설명
- 과목명 / 발표자 / 날짜

**시각 요소**
- 작은 개념 다이어그램 또는 단순한 캘린더+운영 아이콘

### Slide 2. 문제 정의
**목적**
- 왜 이 프로젝트가 필요한지 설득하기

**핵심 메시지**
- 일정 관리의 진짜 문제는 계획 수립보다 계획 유지와 재조정에 있다.

**들어갈 내용**
- 기존 캘린더/투두 앱의 한계 2~3개
- 사용자가 일정이 밀릴 때 겪는 운영 피로

**시각 요소**
- `Before / After` 또는 `기록 앱 vs 운영 앱` 비교

### Slide 3. 핵심 사용자와 시나리오
**목적**
- 누가 이 문제를 가장 강하게 겪는지 보여주기

**핵심 메시지**
- 이 프로젝트는 목표는 많지만 현실 제약 때문에 계획이 자주 무너지는 개인 사용자에게 맞춰져 있다.

**들어갈 내용**
- 핵심 페르소나 3명
- 하루 구조 예시 3개, 주간 구조 예시 3개
- 대표 실패 시나리오 5개

**시각 요소**
- 인물 카드 + 하루 타임라인 + 주간 타임라인

### Slide 4. 제안 솔루션
**목적**
- 문제에 대한 답을 한 장으로 설명하기

**핵심 메시지**
- 목표, 루틴, 외부 일정, 인박스를 모아 AI가 오늘 계획과 재조정안을 제안한다.

**들어갈 내용**
- 입력
- 처리
- 출력

**시각 요소**
- 4~5단계 플로우 다이어그램

### Slide 5. 핵심 기능
**목적**
- 무엇을 실제로 만들 것인지 보여주기

**핵심 메시지**
- 이 프로젝트는 복잡한 범용 플랫폼이 아니라, 일정 운영에 필요한 최소 핵심 기능으로 구성된다.

**들어갈 내용**
- 목표 계획
- 루틴/외부 일정 수집
- 오늘 워크스페이스
- AI 개입

**시각 요소**
- 2x2 기능 카드

### Slide 6. UI/UX 방향과 메인 화면 목업
**목적**
- 실제 사용자 경험의 중심을 보여주기

**핵심 메시지**
- 메인 화면은 기록 화면이 아니라 실행과 복구를 돕는 `오늘 워크스페이스`다.

**들어갈 내용**
- UX 원칙 3개 이하
- 메인 목업 1장
- 목업 위 2~3개의 짧은 주석

**시각 요소**
- 목업 화면 크게
- 좌측 또는 상단에 아주 짧은 원칙 설명

### Slide 7. Spring Boot 구현 구조
**목적**
- 수업 프로젝트로서 구현 가능성과 기술 구조를 설명하기

**핵심 메시지**
- 단일 Spring Boot 애플리케이션 안에서 인증, 일정 엔진, 외부 연동, AI adapter를 분리해 구현한다.

**들어갈 내용**
- Web
- Spring Boot Core
- DB
- External API

**시각 요소**
- 단순 4단 아키텍처 블록

### Slide 8. 중간 진행 상황과 남은 일정
**목적**
- 지금 어디까지 왔는지와 다음 계획을 보여주기

**핵심 메시지**
- 기획과 UX 기준은 정리되었고, 다음 단계는 Spring Boot 구현과 데모 흐름 연결이다.

**들어갈 내용**
- 완료된 것
- 다음 구현 항목
- 최종 발표 데모 목표

**시각 요소**
- 진행 현황 2열 비교 + 4주 타임라인

## 6. 디자인 철학

### 6.1 Simple is Best
- 발표 자료는 기획 문서보다 더 단순해야 한다.
- 정보를 많이 넣는 것보다, 발표자가 말로 보완할 수 있도록 핵심만 남기는 편이 낫다.

### 6.2 인지 부하 이론 적용
- working memory를 많이 요구하는 슬라이드는 피한다.
- 한 슬라이드에서 동시에 비교해야 하는 객체는 최대 2개까지만 둔다.
- 장식성보다 정보 선택과 그룹화가 중요하다.
- 같은 계층의 정보는 같은 모양, 같은 간격, 같은 색을 유지한다.

### 6.3 F-패턴 배치 원칙
- 제목은 항상 좌상단 고정
- 핵심 문장은 제목 바로 아래 1줄 또는 2줄
- 본문은 좌측에서 아래로 짧게 읽히게 구성
- 큰 시각 요소는 우측 또는 하단에 배치
- 사용자의 시선은 `제목 -> 요약 -> 본문 -> 시각 요소` 순서로 흘러야 한다

### 6.4 보고서형 단정함
- 화려한 다크 글로우, 과도한 카드 중첩, 강한 채도 대비를 피한다.
- “사내 보고”처럼 차분하고 정갈한 인상을 우선한다.
- 목업 슬라이드도 제품 홍보용 포스터처럼 만들지 않는다.

## 7. 디자인 팔레트

### 7.1 색상 사용 원칙
- 전체 슬라이드에서 쓰는 핵심 색은 최대 5개로 제한한다.
- 기본은 `배경 1`, `텍스트 2`, `강조 1`, `보조 강조 1` 구조다.
- 같은 의미의 요소는 항상 같은 색을 사용한다.

### 7.2 제안 팔레트
| 역할 | 색상명 | HEX | 사용 용도 |
| --- | --- | --- | --- |
| 배경 | Soft Ivory | `F7F8FA` | 기본 슬라이드 배경 |
| 본문/제목 | Deep Navy | `1F2A37` | 제목, 본문, 주요 라인 |
| 보조 텍스트 | Cool Gray | `6B7280` | 부제, 설명, 캡션 |
| 주 강조 | Muted Teal | `2F6B7A` | 섹션 태그, 핵심 포인트, 링크성 강조 |
| 보조 강조 | Soft Amber | `C89B5C` | 주의, 포인트 숫자, 제한적 강조 |

### 7.3 색상 운용 규칙
- 배경은 거의 항상 `Soft Ivory`
- 제목과 본문은 거의 항상 `Deep Navy`
- 설명이나 캡션은 `Cool Gray`
- 강조는 기본적으로 `Muted Teal`만 사용
- `Soft Amber`는 정말 강조할 숫자, 리스크, 중요 메시지에만 제한적으로 사용

### 7.4 금지 사항
- 슬라이드마다 다른 강조색 사용 금지
- 네온 톤, 형광색, 고채도 블루/퍼플 사용 금지
- 본문에 강조색 다중 사용 금지

## 8. 타이포그래피 기준

### 8.1 폰트 원칙
- 한글 가독성이 좋은 기본 sans-serif만 사용
- 제목과 본문을 다른 개성 강한 폰트로 나누지 않는다
- 발표 자료이므로 “디자인 폰트”보다 “읽히는 폰트”를 우선한다

### 8.2 권장 폰트
- 기본: `맑은 고딕`
- 대체: `Malgun Gothic`, `Noto Sans KR`

### 8.3 크기 기준
| 요소 | 권장 크기 |
| --- | --- |
| 메인 제목 | 28~32pt |
| 슬라이드 부제 | 13~16pt |
| 카드 제목 | 16~18pt |
| 본문 | 14~16pt |
| 캡션/주석 | 10~12pt |

### 8.4 타이포 규칙
- 본문은 좌측 정렬
- 제목은 가능하면 1줄, 최대 2줄
- 한 문장은 짧게 끊고, 한 줄이 너무 길어지지 않게 유지
- Bold는 제목과 핵심 단어에만 쓴다

## 9. 레이아웃 규칙

### 9.1 기본 레이아웃
- 상단 20%: 제목 + 한 줄 요약
- 좌측 35~45%: 핵심 메시지 또는 짧은 본문
- 우측 55~65%: 다이어그램, 목업, 비교표
- 하단: 최소한의 페이지 번호 또는 보조 캡션만

### 9.2 간격 규칙
- 좌우 기본 마진은 넉넉하게 유지
- 텍스트 블록끼리의 간격은 일정하게 고정
- 슬라이드 안에서 카드가 4개를 넘지 않게 유지

### 9.3 슬라이드별 레이아웃 템플릿
- 템플릿 A: `좌측 텍스트 + 우측 시각 요소`
- 템플릿 B: `상단 요약 + 하단 2열 카드`
- 템플릿 C: `상단 제목 + 중앙 플로우 다이어그램`

## 10. 목업 슬라이드 전용 규칙
- 목업은 슬라이드의 주인공이어야 한다.
- 목업 이미지는 전체 슬라이드 너비의 55~65%를 차지하게 한다.
- 목업 옆 설명은 3개 포인트 이내로 제한한다.
- 목업 아래 캡션은 한 줄만 둔다.
- 목업 위에 작은 주석 2~3개 정도는 가능하지만, 텍스트 블록처럼 길어지면 안 된다.

## 11. 리디자인 단계에서 반드시 지킬 체크리스트
- 각 슬라이드가 하나의 질문에만 답하는가
- 제목만 읽어도 발표 흐름이 이어지는가
- 한 슬라이드 안에 본문 포인트가 3개를 넘지 않는가
- 제목, 본문, 시각 요소의 위계가 분명한가
- 팔레트가 5색 안에서 유지되는가
- 다크톤 장식이나 과도한 상자 배치 없이 단정한 보고서형 톤을 유지하는가
- 목업이 설명보다 작지 않은가
