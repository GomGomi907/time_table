# AI 스케줄 오퍼레이터 리부트 기능 정의서

## 1. 제품 기준

### 1.1 제품 목적
- 이 제품은 혼자 일하고 공부하는 하드유저를 위한 `AI 스케줄 오퍼레이터`다.
- 핵심 가치는 일정을 적는 데 있지 않다. 현실 변화로 무너진 일정을 다시 운영 가능한 상태로 되돌리는 데 있다.
- 사용자는 목표, 루틴, 외부 일정, 해야 할 일을 넣고, 시스템은 그 제약을 바탕으로 오늘의 실행 계획을 만들고 조정한다.

### 1.2 핵심 문제
- 일반 캘린더는 약속을 기록하는 데 강하지만, 계획이 밀렸을 때 무엇을 줄이고 무엇을 미뤄야 하는지 판단해주지 않는다.
- 투두 앱은 할 일을 모아두지만, 언제 실행할지와 무엇을 먼저 포기해야 하는지까지 책임지지 않는다.
- 사용자는 일정이 어그러질수록 다시 계획하는 비용이 커지고, 그때마다 현재 집중도까지 잃는다.

### 1.3 1차 사용자
- 혼자 쓰는 개인 파워유저
- 정해진 생활 루틴과 외부 일정이 있고, 그 사이에 공부, 작업, 사이드 프로젝트를 배치해야 하는 사람
- 하루 계획보다 `주간 목표 -> 오늘 실행 -> 실시간 재조정`이 더 중요한 사람

### 1.4 차별점
- `FixedEvent`와 `RoutineTemplate`을 단순 참고 정보가 아니라 일정 생성의 강한 제약으로 다룬다.
- `Goal`과 `InboxItem`을 실제 시간 블록인 `ScheduledBlock`으로 연결한다.
- 앱의 메인 허브는 기록 화면이 아니라 실행 화면인 `Workspace`다.
- AI는 채팅 보조가 아니라 `AgentIntervention`을 통해 재조정 제안을 하는 운영자 역할을 맡는다.

### 1.5 v1 원칙
- 단일 사용자 기준으로 설계한다.
- 메인 허브는 `오늘 워크스페이스` 하나다.
- Google Calendar 연동은 사실상 핵심 입력원이지만, 실패 시 수동 `FixedEvent` 입력으로 우회 가능해야 한다.
- 사용자 승인 없는 완전 자동 스케줄 변경은 기본 정책으로 허용하지 않는다.
- 통계 대시보드, 팀 협업, 위젯 마켓플레이스, 범용 AI 비서는 v1 범위에서 제외한다.

## 2. 핵심 인터페이스

### 2.1 canonical objects
- `Goal`
  - 장기 목표와 주간 목표를 포함한다.
  - 우선순위, 기대 진척, 연결된 오늘 후보 작업을 가진다.
- `RoutineTemplate`
  - 반복되는 생활/근무/이동/회복 제약이다.
  - 고정 루틴인지 유연 루틴인지 표시할 수 있다.
- `FixedEvent`
  - Google Calendar 또는 수동 입력으로 들어온 이동 어려운 일정이다.
  - 일정 생성 시 기본적으로 보호되는 시간 블록이다.
- `InboxItem`
  - 아직 시간에 배정되지 않은 작업이다.
  - 제목, 예상 소요 시간, 관련 `Goal`, 긴급도, 중요도를 가진다.
- `ScheduledBlock`
  - 실제 시간표 위에 배치된 실행 단위다.
  - 출처는 `Goal`, `InboxItem`, `RoutineTemplate`, `FixedEvent` 중 하나 이상일 수 있다.
- `FocusSession`
  - 현재 진행 중인 `ScheduledBlock`의 실행 상태다.
  - 남은 시간, 다음 행동, 일시정지 여부, 개입 필요성을 포함한다.
- `AgentIntervention`
  - AI가 감지한 변화와 그에 대한 재조정 제안이다.
  - 트리거, 제안 요약, 수정 범위, 사용자 선택지를 가진다.
- `RescheduleDecision`
  - 사용자가 `AgentIntervention`에 대해 내린 결정이다.
  - 수락, 수정 수락, 보류, 거절 중 하나다.
- `Workspace`
  - 포커스 보드, 오늘 타임라인, 인박스 큐, 목표 요약, 개입 피드를 담는 실행 공간이다.

### 2.2 최소 필드 기준
| 객체 | v1 최소 필드 |
| --- | --- |
| `Goal` | `id`, `title`, `level`, `priority`, `status`, `parent_goal_id?`, `weekly_target?` |
| `RoutineTemplate` | `id`, `title`, `recurrence_rule`, `start_time`, `end_time`, `rigidity` |
| `FixedEvent` | `id`, `source`, `title`, `start_at`, `end_at`, `calendar_id?`, `is_manual` |
| `InboxItem` | `id`, `title`, `estimate_minutes`, `urgency`, `importance`, `related_goal_id?`, `state` |
| `ScheduledBlock` | `id`, `title`, `source_type`, `source_id`, `goal_id?`, `start_at`, `end_at`, `status` |
| `FocusSession` | `id`, `scheduled_block_id`, `state`, `started_at?`, `remaining_minutes`, `is_paused` |
| `AgentIntervention` | `id`, `trigger_type`, `summary`, `option_set`, `affected_block_ids`, `expires_at?`, `state` |
| `RescheduleDecision` | `id`, `intervention_id`, `choice`, `patch_summary`, `decided_at` |
| `Workspace` | `date`, `active_focus_session_id?`, `timeline_block_ids`, `inbox_item_ids`, `goal_summary_ids`, `intervention_feed_ids` |

### 2.3 관계 규칙
- 모든 `ScheduledBlock`은 적어도 하나의 출처를 가져야 한다.
- `FixedEvent`는 기본적으로 다른 블록보다 우선한다.
- `Goal`은 직접 시간표에 올라가지 않고, `InboxItem` 또는 `ScheduledBlock`을 통해 실행된다.
- `AgentIntervention`은 기존 블록을 바꾸기 전에 반드시 사용자에게 `RescheduleDecision`을 요청한다.
- `Workspace`와 `목표 계획` 페이지는 동일한 `Goal`, `InboxItem`, `ScheduledBlock` 소스를 공유한다.

## 3. 기능 정의

### 3.1 목표 설계 및 주간 목표 분해
**목적**
- 장기 방향과 이번 주 실행 우선순위를 구조화해 일정 생성 기준을 만든다.

**사용자 문제**
- 사용자는 오늘 바쁜 일정을 처리하다 보면 왜 이 일을 해야 하는지 잊기 쉽다.
- 장기 목표가 오늘 시간표와 연결되지 않으면 일정은 금방 긴급한 일 위주로 무너진다.

**입력**
- 장기 `Goal` 제목과 설명
- 이번 주 `Goal` 제목, 우선순위, 기대 진척
- 오늘 후보 작업과 예상 소요 시간

**상태**
- `draft`
- `active`
- `paused`
- `completed`

**핵심 규칙**
- 모든 주간 목표는 상위 장기 목표와 연결되어야 한다.
- 오늘 후보 작업은 주간 목표에서 파생되거나, 주간 목표와 직접 연결 가능한 `InboxItem`이어야 한다.
- 목표 우선순위는 일정 배치 순서에 영향을 주지만 `FixedEvent`를 침범하지 않는다.
- 주간 목표가 과도하게 크면 시스템은 분해를 요구하고 바로 스케줄링하지 않는다.

**성공 조건**
- 사용자가 오늘 배정된 블록이 어떤 목표를 위한 것인지 즉시 알 수 있다.
- 주간 목표 단위로 이번 주 남은 실행량을 설명할 수 있다.

**예외 상황**
- 상위 목표 없이 주간 목표만 입력된 경우: 임시 독립 목표로 저장하되 추후 연결 필요 상태로 표시한다.
- 목표가 너무 넓어 실행 단위가 없는 경우: 일정 배정 대신 분해 요청으로 되돌린다.

**v1 포함 범위**
- 장기 목표, 주간 목표, 오늘 후보 작업 정의
- 목표 우선순위와 일정 생성 연결
- 목표 기반 오늘 추천 작업 생성

### 3.2 루틴/고정 일정 수집
**목적**
- 사용자의 반복 제약과 외부 약속을 모아 스케줄 가능한 시간을 계산한다.

**사용자 문제**
- 실제로는 비어 있지 않은 시간을 계획 가능한 시간으로 착각해 무리한 계획을 세운다.
- 외부 일정과 생활 루틴이 빠지면 AI 제안이 현실과 어긋난다.

**입력**
- `RoutineTemplate`
- Google에서 가져온 `FixedEvent`
- 수동 `FixedEvent`

**상태**
- 루틴: `draft`, `active`, `archived`
- 연동 일정: `connected`, `syncing`, `degraded`, `error`

**핵심 규칙**
- `FixedEvent`는 기본적으로 보호되는 시간이다.
- `RoutineTemplate`은 고정 또는 유연으로 표시할 수 있다.
- Google 연동 실패 시 수동 `FixedEvent` 입력으로 우회 가능해야 한다.
- 동일 시간대 중복 외부 일정은 중복 감지 후 병합 또는 사용자 확인을 요구한다.

**성공 조건**
- 사용 가능한 시간과 보호 시간의 경계가 분명해진다.
- 일정 초안 생성 시 이미 확보된 제약 조건을 다시 물어보지 않는다.

**예외 상황**
- Google 연결이 없거나 끊긴 경우: 수동 일정 입력 흐름을 바로 제공한다.
- 하루 전체가 `FixedEvent`와 루틴으로 막힌 경우: 일정 초안 대신 용량 부족 경고를 낸다.

**v1 포함 범위**
- Google Calendar 기반 `FixedEvent` 수집
- 수동 `FixedEvent` 추가
- 반복 루틴 입력과 수정

### 3.3 인박스 수집 및 정리
**목적**
- 아직 시간에 올리지 않은 작업을 빠르게 모으고, 스케줄 가능한 재료로 정리한다.

**사용자 문제**
- 해야 할 일이 떠오를 때 즉시 기록하지 못하면 잊어버린다.
- 기록만 되고 정리가 안 되면 AI가 어떤 작업을 먼저 배치해야 할지 판단하기 어렵다.

**입력**
- 빠른 입력
- 목표에서 파생된 후보 작업
- 자연어 메모에서 추출한 작업

**상태**
- `captured`
- `clarified`
- `ready`
- `snoozed`
- `archived`

**핵심 규칙**
- `InboxItem`은 기본적으로 시간 미배정 상태다.
- v1에서는 제목과 예상 소요 시간은 필수다.
- 관련 `Goal`이 없더라도 저장은 가능하지만, 일정 추천 우선순위는 낮아진다.
- `ready` 상태가 된 항목만 AI 일정 초안 생성에 적극 반영한다.

**성공 조건**
- 사용자가 작업을 10초 이내에 인박스에 넣을 수 있다.
- 스케줄링 가능한 작업과 아직 모호한 작업이 분리된다.

**예외 상황**
- 소요 시간이 비어 있는 경우: 기본 추정치로 저장하되 재정리 필요 상태로 표시한다.
- 같은 제목의 작업이 반복 저장된 경우: 중복 후보로 표시하고 병합 선택지를 준다.

**v1 포함 범위**
- 빠른 인박스 입력
- 간단한 정리 상태 전환
- 목표 연결과 예상 소요 시간 부여

### 3.4 AI 기반 일정 초안 생성
**목적**
- 수집된 목표, 제약, 인박스 작업을 바탕으로 오늘 실행 가능한 계획을 제안한다.

**사용자 문제**
- 계획을 세우는 데 시간이 많이 들고, 무엇을 포기해야 하는지 판단하기 어렵다.
- 목표는 많은데 실제 가용 시간이 적을 때 손으로 우선순위를 재배치하는 비용이 크다.

**입력**
- `Goal`
- `RoutineTemplate`
- `FixedEvent`
- `InboxItem`
- 작업 시간 선호도와 정책 설정

**상태**
- `not_generated`
- `draft_ready`
- `reviewed`
- `accepted`
- `manually_adjusted`

**핵심 규칙**
- 초안은 오늘 기준으로 먼저 생성한다.
- `FixedEvent`와 고정 루틴은 침범하지 않는다.
- 가용 시간이 부족하면 일부 작업은 인박스에 남기고, 남긴 이유를 명시한다.
- 각 `ScheduledBlock`은 왜 그 시간에 배치되었는지 설명 가능해야 한다.
- 사용자가 수락한 초안을 시스템이 몰래 다시 덮어쓰지 않는다.

**성공 조건**
- 사용자가 오늘 시작 전 1회 검토만으로 실행 가능한 시간표를 얻는다.
- 일정 초안이 목표 우선순위와 제약을 함께 반영한다.

**예외 상황**
- 입력 데이터가 부족한 경우: 생성 대신 필요한 최소 입력을 안내한다.
- 외부 일정 동기화 실패로 제약이 불완전한 경우: 불완전 상태를 표시하고 수동 검토를 요구한다.

**v1 포함 범위**
- 오늘 기준 일정 초안 생성
- 초안 수락 또는 재생성
- 남긴 작업에 대한 사유 노출

### 3.5 AI 개입 및 재조정 제안
**목적**
- 실행 중 일정 충돌, 지연, 빈 시간, 우선순위 변화가 발생했을 때 즉시 새로운 선택지를 제안한다.

**사용자 문제**
- 계획은 시작보다 중간 붕괴가 더 치명적이다.
- 어떤 작업을 연장하고 무엇을 미뤄야 할지 그때그때 판단하기 어렵다.

**입력**
- 지연된 `FocusSession`
- 새로 들어온 `FixedEvent`
- 열린 빈 시간
- 목표 위험 신호
- 사용자의 완료/미완료 피드백

**상태**
- `pending`
- `accepted`
- `edited`
- `dismissed`
- `expired`

**핵심 규칙**
- 주요 트리거는 일정 초과, 새 외부 일정 충돌, 큰 빈 시간 발생, 목표 위험 상태다.
- 제안 종류는 연장, 분할, 이동, 삭제 후보 제안, 보류, 버퍼 삽입으로 제한한다.
- 사용자 선택지는 수락, 수정 수락, 나중에 보기, 거절로 고정한다.
- `RescheduleDecision`이 확정되기 전에는 기존 시간표를 확정 변경하지 않는다.
- 자동 반영 금지 원칙을 유지한다. v1 기본 정책에서 AI는 승인 없이 `ScheduledBlock`을 재배치하지 않는다.

**성공 조건**
- 일정이 틀어졌을 때 사용자가 다음 행동을 1분 안에 결정할 수 있다.
- 변경 이력과 변경 이유가 남아 추후 재검토 가능하다.

**예외 상황**
- 사용자가 여러 개입을 연속으로 무시한 경우: 우선순위를 낮추고 묶음 제안으로 전환한다.
- 충돌이 너무 복잡한 경우: 단일 추천안 대신 선택 가능한 2~3개 플랜을 제시한다.

**v1 포함 범위**
- 주요 트리거 감지
- 단일 또는 소수 플랜 제안
- 승인 기반 재조정

### 3.6 오늘 워크스페이스와 포커스 실행
**목적**
- 사용자가 지금 해야 할 일과 다음 조치를 한 화면에서 실행한다.

**사용자 문제**
- 실행 중에는 설정과 계획 화면을 오가면 집중이 깨진다.
- 현재 작업, 다음 작업, 변경 제안이 분리되어 있으면 판단 비용이 커진다.

**입력**
- 오늘의 `ScheduledBlock`
- 현재 `FocusSession`
- 인박스 대기 작업
- 목표 요약
- 최근 `AgentIntervention`

**상태**
- `idle`
- `active`
- `paused`
- `needs_decision`
- `day_closed`

**핵심 규칙**
- 한 번에 하나의 `FocusSession`만 1차 포커스로 노출한다.
- 메인 화면은 포커스 보드, 오늘 타임라인, 인박스 큐, 목표 요약, 개입 피드로 구성한다.
- 완료, 연장, 보류, 건너뛰기 액션은 모두 여기서 실행 가능해야 한다.
- 포커스 보드는 현재 블록의 의미를 `Goal` 기준으로 설명해야 한다.

**성공 조건**
- 사용자가 실행 중 대부분의 결정을 `Workspace` 밖으로 나가지 않고 내릴 수 있다.
- 현재 할 일과 다음 변화가 같은 화면에서 보인다.

**예외 상황**
- 오늘 일정이 아직 없는 경우: 초안 생성 CTA를 먼저 보여준다.
- 현재 블록이 없는 경우: 다음 블록 또는 빈 시간 활용 제안을 보여준다.

**v1 포함 범위**
- 포커스 보드
- 오늘 타임라인
- 인박스 큐
- 목표 요약 패널
- 개입 피드

### 3.7 목표와 일정의 연결
**목적**
- `Goal`이 실제 실행과 분리되지 않도록 일정, 인박스, 포커스 상태를 연결한다.

**사용자 문제**
- 목표가 보기 좋은 계획으로만 남고, 실제 시간 사용과 연결되지 않는 경우가 많다.
- 일정이 바뀌면 목표 진척이 어떻게 흔들리는지 바로 알기 어렵다.

**입력**
- `Goal`
- `InboxItem`
- `ScheduledBlock`
- 완료/보류 기록

**상태**
- `linked`
- `unlinked`
- `on_track`
- `at_risk`

**핵심 규칙**
- `ScheduledBlock`은 가능하면 하나의 대표 `Goal`과 연결되어야 한다.
- 완료된 블록은 목표 진척 갱신의 근거가 되지만, v1에서는 완전 자동 퍼센트 계산보다 상태 보조 정보에 가깝게 다룬다.
- 목표 위험 신호가 감지되면 `AgentIntervention` 생성 근거가 된다.
- `목표 계획` 페이지와 `Workspace`의 목표 요약은 동일한 연결 상태를 보여줘야 한다.

**성공 조건**
- 사용자가 오늘 무엇을 놓치면 어떤 목표가 위험해지는지 이해할 수 있다.
- 목표 계획과 실행 화면의 문맥이 끊기지 않는다.

**예외 상황**
- 관련 목표가 없는 작업이 많은 경우: 독립 작업 묶음으로 관리하되 목표 연결 추천을 제공한다.
- 하나의 블록이 여러 목표에 걸친 경우: 대표 목표 1개를 선택하고 보조 태그만 허용한다.

**v1 포함 범위**
- 목표-작업-시간블록 연결
- 목표 위험 상태 표시
- 워크스페이스 요약 반영

### 3.8 Google Calendar 연동
**목적**
- 외부 약속을 자동 수집해 현실적인 일정 초안과 재조정을 가능하게 한다.

**사용자 문제**
- 이미 잡혀 있는 회의나 개인 약속을 수동 입력하면 누락이 많고 관리 비용이 높다.
- 외부 일정 누락은 AI 제안의 신뢰도를 크게 떨어뜨린다.

**입력**
- Google OAuth 연결
- 동기화 대상 캘린더 선택
- 동기화 시점과 범위

**상태**
- `not_connected`
- `connected`
- `syncing`
- `degraded`
- `error`

**핵심 규칙**
- v1에서는 Google Calendar를 우선 연동 대상으로 삼는다.
- 가져온 일정은 기본적으로 `FixedEvent`로 취급한다.
- 동기화 오류가 발생하면 마지막 성공 시점과 누락 가능성을 명시한다.
- 사용자가 연동에 실패해도 수동 `FixedEvent`로 동일 기능 흐름을 계속 진행할 수 있어야 한다.

**성공 조건**
- 사용자의 핵심 외부 일정이 오늘 시간표 생성 전에 반영된다.
- 연동 상태가 신뢰 가능하게 노출된다.

**예외 상황**
- 권한 거부 또는 토큰 만료: 재연결 흐름과 수동 입력 fallback을 함께 노출한다.
- 캘린더가 너무 많거나 노이즈가 많은 경우: 포함할 캘린더를 제한하도록 한다.

**v1 포함 범위**
- 단일 Google 계정 연결
- 캘린더 일정 가져오기
- 연동 오류 및 재연결 처리

### 3.9 설정, 선호도, 개입 정책 관리
**목적**
- 일정 생성과 AI 개입이 사용자의 선호와 현실에 맞도록 조정한다.

**사용자 문제**
- 같은 일정 충돌도 사람마다 원하는 대응이 다르다.
- 개입 빈도, 버퍼 선호, 집중 시간대가 반영되지 않으면 제안이 피로해진다.

**입력**
- 작업 시간대 선호
- 버퍼 삽입 기준
- 개입 빈도와 조용한 시간
- 재조정 우선순위 기준

**상태**
- `default`
- `customized`
- `reset_pending`

**핵심 규칙**
- 정책은 일정 초안 생성과 `AgentIntervention` 판단에 즉시 반영된다.
- 사용자가 이해할 수 없는 숨은 규칙을 두지 않는다.
- 조용한 시간대에는 고위험 개입만 허용한다.
- 설정 변경은 이후 생성과 개입부터 적용하고, 이미 확정된 블록은 사용자 확인 없이 변경하지 않는다.

**성공 조건**
- 사용자가 AI 제안이 왜 그렇게 나왔는지 설정 기준으로 설명받을 수 있다.
- 개입 피로도와 일정 추천 품질이 함께 관리된다.

**예외 상황**
- 설정 충돌이 있는 경우: 우선순위 높은 정책만 적용하고 나머지는 사용자 확인을 요구한다.
- 지나치게 제한적인 설정으로 일정 생성이 불가능한 경우: 생성 실패 이유와 완화 제안을 함께 보여준다.

**v1 포함 범위**
- 개입 빈도
- 조용한 시간
- 버퍼 정책
- 일정 재조정 우선순위

## 4. v1 제외 범위
- 팀 협업, 공유 워크스페이스
- 위젯 스토어, 커뮤니티 프리셋
- 고급 통계 리포트와 성과 대시보드
- 다중 캘린더 생태계 확장
- 승인 없는 완전 자동 재조정
