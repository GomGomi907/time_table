# AI 스케줄 오퍼레이터 리부트 UI/UX 설계서

## 1. 문서 목적
- 이 문서는 `reboot_persona.md`, `reboot_feature_spec.md`, `reboot_page_spec.md`를 실제 화면 경험으로 번역하는 UI/UX 기준 문서다.
- 목적은 예쁜 대시보드를 만드는 것이 아니라, 사용자가 `계획 -> 실행 -> 개입 -> 재조정` 흐름을 한 번에 이해하고 움직일 수 있게 하는 것이다.
- 이번 문서는 v1 핵심 화면인 `오늘 워크스페이스`를 우선 설계하고, 나머지 페이지는 그 경험을 보조하는 방식으로 정리한다.

## 2. UX 핵심 원칙

### 2.1 Execution First
- 메인 허브는 계획 화면이 아니라 실행 화면이어야 한다.
- 첫 시선은 항상 현재 `FocusSession`으로 가야 한다.
- 사용자가 오늘 해야 할 일을 찾기 위해 여러 패널을 탐색하게 두지 않는다.

### 2.2 Constraint Visible
- 시간 제약은 숨은 로직이 아니라 화면에서 보이는 구조여야 한다.
- `FixedEvent`, 보호 시간, 밀린 블록, 비어 있는 시간은 시각적으로 구분되어야 한다.

### 2.3 Explainable AI
- AI는 “추천”이 아니라 “추천 이유가 있는 운영자”로 보이게 해야 한다.
- 모든 `AgentIntervention`은 이유, 영향 범위, 선택지 3가지를 함께 보여야 한다.

### 2.4 Low Cognitive Load
- 한 화면에 많은 정보를 두더라도, 한 순간에 한 결정만 요구해야 한다.
- 정보는 많아도 시선의 우선순위는 명확해야 한다.
- 장식보다 정보 위계, 상태 구분, 실행 버튼 위치가 더 중요하다.

### 2.5 Recovery over Perfection
- 사용자의 하루는 늘 무너질 수 있다는 전제로 설계한다.
- 빈 상태, 실패 상태, 연동 오류, 일정 충돌은 “복구 경로”를 먼저 보여줘야 한다.

### 2.6 Controlled Automation
- AI가 제안은 많이 해도, 사용자의 승인 없이 확정 변경하지 않는다.
- 사용자는 언제든 제안을 수정하거나 미룰 수 있어야 한다.

## 3. 경험 구조

### 3.1 핵심 사용자 루프
1. 외부 일정과 루틴을 불러온다.
2. 목표와 인박스를 오늘 후보 작업으로 정리한다.
3. AI가 오늘 초안을 제안한다.
4. 사용자는 `오늘 워크스페이스`에서 실행한다.
5. 일정이 흔들리면 `AgentIntervention`이 개입한다.
6. 사용자는 빠르게 결정하고 다시 실행으로 복귀한다.

### 3.2 화면 우선순위
- 1순위: 오늘 워크스페이스
- 2순위: 목표 계획
- 3순위: 온보딩/초기 설정
- 4순위: 설정·연동

### 3.3 전역 표면 우선순위
- 1순위: AI 개입 팝업/패널
- 2순위: 빠른 작업 추가 입력
- 3순위: 세션 종료/일정 변경 확인 모달

## 4. 시각 방향

### 4.1 디자인 키워드
- Control room
- Calm urgency
- Dense but legible
- Tactical clarity

### 4.2 비주얼 방향
- 기존의 장식형 글래스모피즘보다 정보 운영실에 가까운 밀도 있는 화면을 지향한다.
- 지나치게 차갑거나 과한 SF 톤 대신, 짙은 잉크색 바탕 위에 따뜻한 경고색과 청록 계열 포인트를 섞는다.
- “AI가 대신해 주는 마법”이 아니라 “상황판을 보며 결정을 돕는 운영 도구”처럼 보여야 한다.

### 4.3 컬러 역할
| 역할 | 방향 |
| --- | --- |
| 기본 배경 | 짙은 슬레이트/잉크 계열 |
| 주요 텍스트 | 고명도 아이보리/웜 그레이 |
| 강조 색 | 청록, 코랄, 앰버 |
| 성공 | 선명한 민트 |
| 위험 | 오렌지-코랄 |
| 보호 시간 | 차분한 블루-그레이 |

### 4.4 타이포그래피 원칙
- 헤드라인은 현재 작업과 상태를 즉시 읽을 수 있도록 크고 짧게 쓴다.
- 보조 텍스트는 설명이 아니라 판단 보조 역할만 한다.
- 숫자와 시간은 시각적으로 안정적인 tabular 성격이 느껴져야 한다.

## 5. 레이아웃 시스템

### 5.1 데스크톱 구조
- 좌측: 전역 네비게이션과 오늘 상태 요약
- 중앙: 포커스 보드와 오늘 타임라인
- 우측: AI 개입 카드, 목표 요약, 동기화 상태
- 하단: 인박스 큐와 최근 개입 피드

### 5.2 모바일 구조
- 상단 고정: 현재 `FocusSession`
- 중단: 오늘 타임라인
- 하단 탭 또는 아코디언: 목표 요약, 인박스, 개입 피드
- AI 개입은 전체 너비 하단 시트로 제공

### 5.3 정보 위계
1. 지금 해야 할 일
2. 지금 바뀐 것 또는 곧 바뀔 것
3. 오늘 남은 시간 구조
4. 이 작업이 연결된 목표
5. 아직 배정하지 않은 작업

## 6. 핵심 화면 설계

### 6.1 오늘 워크스페이스
**역할**
- 사용자가 가장 오래 머무는 화면
- 실행, 판단, 복구가 모두 이 안에서 가능해야 함

**핵심 블록**
- 포커스 보드
- 오늘 타임라인
- AI 운영 카드
- 목표 벡터 요약
- 인박스 큐
- 개입 히스토리

**핵심 UX 규칙**
- 포커스 보드는 화면 중앙에서 가장 먼저 보이게 한다.
- AI 운영 카드는 포커스 보드를 가리지 않지만, 결정이 필요할 때 즉시 눈에 들어와야 한다.
- 타임라인은 “전체 계획”이 아니라 “오늘 남은 운용 상태”를 읽게 해야 한다.
- 목표 요약은 동기부여 카드가 아니라 현재 작업의 의미를 연결하는 패널이어야 한다.

### 6.2 목표 계획
**역할**
- 목표를 관리하는 화면이 아니라, 오늘 후보 작업을 만드는 편집실

**UX 규칙**
- 장기 목표, 주간 목표, 오늘 후보 작업이 한 흐름으로 이어져야 한다.
- 목표를 생성하는 순간 바로 오늘 시간표와 연결 가능성이 보여야 한다.

### 6.3 온보딩/초기 설정
**역할**
- 사용자의 현실 제약과 선호를 빠르게 수집하는 입력 허브

**UX 규칙**
- 계정 생성보다 Google 연결과 루틴 수집이 먼저 중요하다.
- 연동 실패 시 대체 경로를 바로 보여줘야 한다.
- 첫 일정 초안 생성까지 끊기지 않는 흐름이어야 한다.

### 6.4 설정·연동
**역할**
- 사용자가 AI를 제어하고 신뢰를 회복하는 화면

**UX 규칙**
- 설정은 많아도 원리 설명 없이 던져두지 않는다.
- 동기화 오류, 개입 정책, 루틴 수정은 각자 이유와 영향 범위를 보여줘야 한다.

## 7. 상호작용 패턴

### 7.1 포커스 세션 액션
- `완료`: 현재 블록 종료 후 다음 블록 또는 빈 시간 제안으로 이동
- `10분 연장`: 현재 블록은 유지하되 뒤 영향 범위를 즉시 프리뷰
- `보류`: 세션은 멈추되, 나중에 복귀 가능한 상태 유지
- `건너뛰기`: 포기라기보다 재배치의 시작으로 취급

### 7.2 AI 개입 패턴
- 개입은 조용히 등장하되, 무시하면 사라지는 광고처럼 보이면 안 된다.
- 카드 안에는 반드시 “문제”, “추천”, “영향”이 분리되어 있어야 한다.
- 추천안은 최대 3개까지만 보여준다.

### 7.3 빠른 입력 패턴
- 사용자는 앱 어디서든 생각난 일을 10초 안에 저장해야 한다.
- 빠른 입력은 기본적으로 `InboxItem` 중심이지만, 고정 시간이 포함되면 `FixedEvent`로 전환할 수 있어야 한다.

### 7.4 목표 연결 패턴
- 현재 작업이 어떤 `Goal`을 위한 것인지 포커스 보드와 타임라인 양쪽에서 확인 가능해야 한다.
- 목표와 연결되지 않은 작업은 의도적으로 낮은 신뢰 상태처럼 보여주는 것이 좋다.

## 8. 상태 디자인

### 8.1 정상 상태
- 오늘 계획이 확정되어 있고, 현재 작업과 다음 블록이 명확한 상태

### 8.2 복구 상태
- 일정 초과, 충돌, 누락, 연동 실패 등으로 개입이 필요한 상태
- 이 상태에서는 에러 자체보다 “다음 가능한 행동”을 먼저 보여준다.

### 8.3 빈 상태
- 오늘 일정 없음
- 목표 없음
- 인박스 비어 있음
- 개입 기록 없음
- 각 빈 상태는 “아직 아무것도 없다”가 아니라 “다음으로 무엇을 하면 되는가”를 안내해야 한다.

## 9. 콘텐츠와 마이크로카피

### 9.1 톤
- 짧고 단정해야 한다.
- 위로하는 말보다 운영 판단을 돕는 말이 우선이다.
- “해야 합니다”보다는 “이렇게 바꾸면 어떤 영향이 있다”를 알려준다.

### 9.2 좋은 문장 예시
- “현재 블록이 18분 밀렸습니다.”
- “영어 회화 블록을 22:10으로 이동하면 오늘 목표는 유지됩니다.”
- “캘린더 동기화가 2시간 전 이후 멈췄습니다. 수동 고정 일정을 추가할 수 있습니다.”

### 9.3 피해야 할 문장
- “당신은 충분히 잘하고 있어요.”
- “AI가 완벽한 일정을 다시 만들었어요.”
- “문제가 발생했습니다.”만 보여주고 끝나는 문장

## 10. 접근성과 반응형 기준
- 모든 핵심 상태는 색만으로 구분하지 않는다.
- 시간과 상태, 영향 범위는 텍스트로도 확인 가능해야 한다.
- 버튼은 모바일에서도 최소 44px 터치 타겟을 확보한다.
- 포커스 보드는 작은 화면에서도 가장 먼저 읽히는 구조여야 한다.

## 11. 목업 우선 범위

### 11.1 이번 목업의 대상
- `오늘 워크스페이스` 데스크톱 우선 화면 1종

### 11.2 목업에서 반드시 보여야 할 요소
- 중앙 포커스 보드
- 오늘 타임라인
- AI 운영 카드
- 목표 벡터 요약
- 인박스 큐
- 최근 개입 피드
- Google 동기화 상태

### 11.3 목업에서 표현할 핵심 상태
- 현재 `FocusSession`이 진행 중인 상태
- 연장/이동 판단이 필요한 `AgentIntervention`이 떠 있는 상태
- 목표는 살아 있고, 오늘 시간표와 연결된 상태
- 인박스에 아직 배정되지 않은 작업이 남아 있는 상태

### 11.4 목업 해석 기준
- 이 화면만 보고도 “지금 무엇을 해야 하는지”, “무엇이 문제인지”, “다음으로 무엇을 누르면 되는지”가 읽혀야 한다.
- 단순한 미형보다 정보 운영과 복구 동선이 더 잘 보여야 한다.
