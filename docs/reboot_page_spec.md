# Time_table 페이지별 기획서

업데이트: 2026-04-19

본 문서는 현재 `frontend`와 `backend` 구현 상태를 기준으로 정리한 페이지 기획서다.
과거의 랜딩 중심 문서나 캔버스 실험안보다 이 문서를 우선한다.

## 1. 문서 목적

- 제품의 첫 인상을 소개 페이지가 아니라 작업 화면 중심으로 재정렬한다.
- 각 페이지가 어떤 사용자 질문에 답해야 하는지 명확히 한다.
- 페이지별 핵심 콘텐츠, 액션, 상태, 연결 API, 디자인 방향을 한글 기준으로 정리한다.
- 이후 프론트 구현 순서를 정할 수 있는 기준 문서로 사용한다.

## 2. 현재 기준 제품 원칙

### 2.1 제품 성격
- 이 제품은 마케팅 랜딩 사이트가 아니라 개인 운영용 워크스페이스다.
- 첫 진입은 설명보다 현재 상태와 다음 행동을 바로 보여줘야 한다.
- 사용자는 “이 앱이 무엇인가”보다 “지금 무엇을 해야 하는가”를 먼저 본다.

### 2.2 홈 정책
- `/`는 독립 랜딩 페이지를 두지 않는다.
- `/`는 항상 `/dashboard`로 리다이렉트한다.
- 제품 소개성 카피는 별도 페이지를 만들더라도 운영 화면을 방해하지 않는 범위에서만 둔다.

### 2.3 주력 구조
- 주력 백엔드는 Spring Boot다.
- LLM 기반 일정 정리는 로컬 `gemma4`를 사용한다.
- Google OAuth는 선택 기능이다. 없어도 핵심 기능은 동작해야 한다.

| 구분 | 경로 | 상태 | 역할 |

## 4. 공통 디자인 원칙

### 4.1 시각 톤
- 배경은 밝은 오프화이트 계열을 유지한다.
- 흰 카드와 옅은 그림자로 정보 레이어를 분리한다.
- 검정 또는 매우 진한 회색을 핵심 텍스트와 주요 CTA에만 쓴다.
- 의미 없는 장식성 그라데이션, 과한 유리 효과, 홍보성 비주얼은 지양한다.

### 4.2 타이포그래피
- 기본 본문은 한글 가독성을 위해 `Noto Sans KR` 중심으로 간다.
- 강조 헤드라인은 `Inter` 계열 display 톤을 제한적으로 사용한다.
- 영문 대문자 라벨은 `metric-label`, `status-chip`, API 태그에만 제한적으로 사용한다.
- 본문 카피는 짧고 명령형으로 쓴다. 소개 문장보다 상태 문장을 우선한다.

### 4.3 레이아웃
- 최대 폭은 현재 `page-shell` 기준을 유지한다.
- 페이지 공통 구조는 `상단 상태 요약 + 본문 작업 영역 + 보조 패널` 순서를 기본으로 한다.
- 데스크톱에서는 좌우 분할을 허용하되, 모바일에서는 세로 단일 컬럼으로 자연스럽게 접혀야 한다.

### 4.4 컴포넌트 원칙
- 버튼은 `pill-button`, `ghost-pill` 계열을 유지하되 액션 위계를 명확히 한다.
- 카드 제목은 “설명”보다 “기능명 + 현재 상태”가 먼저 보이게 한다.
- 빈 상태, 로딩 상태, 에러 상태는 모든 카드에서 독립적으로 보여야 한다.
- 하나의 페이지에 너무 많은 CTA를 두지 않는다. 기본 CTA 1개, 보조 CTA 1~2개를 넘기지 않는다.

### 4.5 금지 사항
- 랜딩 페이지식 hero 카피를 루트 진입에 다시 넣지 않는다.
- 페이지를 설명하려고 제품을 설명하는 문단을 길게 쓰지 않는다.
- 영어 카피를 기본값으로 두지 않는다.
- “예뻐 보이는” 카드보다 “지금 결정을 내릴 수 있는” 카드가 우선이다.

## 5. 글로벌 네비게이션 원칙

### 5.1 상단 공통 네비게이션
- 좌측: 현재 페이지명 또는 현재 컨텍스트
- 우측: 문서, 설정, 상태 새로고침 같은 보조 액션
- 전역 검색이나 커맨드바는 현재 범위에서는 보류한다.

### 5.2 주 내비게이션
- 1차 노출 메뉴는 `대시보드`, `목표`, `일정`, `설정` 네 개로 제한한다.
- `API 문서`는 개발용 보조 메뉴로 둔다.
- `온보딩`과 `인증 콜백`은 일반 메뉴에 노출하지 않는다.

### 5.3 전역 표면
- 토스트: 저장 성공, 실패, 재시도 안내
- 모달: 일정 import, 목표 추가, 위험한 설정 변경 확인
- 연결 상태 배너: Google 미설정, 백엔드 불가, gemma4 실행 문제 등을 짧게 표시

## 6. 페이지별 상세 기획

### 6.1 `/` 루트

**페이지 목적**
- 별도 페이지 역할을 하지 않는다.
- 사용자를 가장 중요한 운영 화면으로 보내는 진입점이다.

**반드시 들어가야 할 것**
- 없음

**동작**
- 즉시 `/dashboard`로 리다이렉트

**디자인 방향**
- UI를 만들지 않는다.
- 플래시성 로딩 화면도 가능하면 두지 않는다.

**비고**
- 루트에 랜딩 성격의 콘텐츠를 다시 넣는 것은 금지한다.

### 6.2 `/dashboard` 메인 대시보드

**페이지 목적**
- 사용자가 앱에 들어오자마자 이번 주 일정, 목표 진행, Google 연결 상태를 한 번에 파악하게 한다.
- “설정하는 곳”이 아니라 “운영하는 곳”이어야 한다.

**핵심 사용자 질문**
- 이번 주 일정이 어떻게 구성되어 있는가
- 방금 import한 일정이 제대로 반영됐는가
- Google 연결은 가능한가
- 지금 추적 중인 목표는 무엇인가

**반드시 들어가야 할 영역**
- 상단 상태 요약 헤더
  - 백엔드 상태
  - 로컬 Gemma 4 상태
  - Google 연결 가능 여부
  - 주간 시간표 존재 여부
- 메인 일정 보드
  - 요일별 주간 일정
  - 새로고침 버튼
  - 일정 import 버튼
  - 로딩, 에러, 빈 상태
- 우측 보조 패널
  - Google 연결 카드
  - 목표 트리 요약 카드
  - 이 페이지가 연결된 API 요약 카드

**핵심 액션**
- 주간 일정 새로고침
- 주간 일정 텍스트 import
- Google 연결 시작
- 목표 추가
- API 문서 열기

**연결 API**
- `GET /api/schedule/week`
- `POST /api/schedule/import`
- `GET /api/goals`
- `POST /api/goals`
- `GET /api/auth/session`
- `GET /api/auth/google/start`

**상태 설계**
- 로딩: 카드별 스켈레톤 또는 로더
- 빈 상태: “아직 일정이 없습니다”, “아직 저장된 목표가 없습니다”
- 에러 상태: 카드 단위 에러와 재시도 버튼
- 부분 실패 허용: 예를 들어 목표 로딩 실패가 일정 보드를 막지 않아야 한다

**디자인 방향**
- 가장 넓은 화면 비중을 일정 보드에 배정한다.
- 우측 패널은 요약과 즉시 액션에 집중한다.
- hero 문구는 소개문이 아니라 현재 화면 설명 수준으로만 둔다.
- 카드 간 색 차이는 의미를 드러내는 범위에서만 사용한다.
- 일정 카드는 카테고리별 톤 차이를 주되 전체 바탕은 밝게 유지한다.

**모바일 규칙**
- 상단 요약 카드는 2열 또는 1열로 자동 접기
- 일정 보드를 먼저 보여주고, Google 연결과 목표는 아래로 내린다
- 우측 패널 위젯은 세로 카드 스택으로 변환

**이 페이지에서 하지 말아야 할 것**
- 긴 제품 소개
- 세부 설정 폼 전체 노출
- 목표 상세 편집의 모든 기능 수용

### 6.3 `/goals` 목표 페이지

**페이지 목적**
- 목표 트리를 위젯 수준이 아니라 실제 관리 도구 수준으로 확장한다.
- 장기 목표, 이번 주 목표, 오늘 후보 작업의 연결을 정리한다.

**핵심 사용자 질문**
- 이번 주에 밀 수 없는 목표는 무엇인가
- 어떤 목표가 현재 위험 상태인가
- 오늘 일정으로 내려보낼 다음 작업은 무엇인가

**반드시 들어가야 할 영역**
- 상단 목표 요약
  - 진행 중 목표 수
  - 위험 목표 수
  - 이번 주 핵심 목표
- 좌측 목표 트리
  - 상위 목표
  - 하위 목표
  - 진행률
  - 상태 태그
- 우측 상세 패널
  - 선택한 목표 설명
  - 연결된 하위 목표
  - 주간 우선순위
  - 일정으로 보내기 후보 액션
- 빠른 추가 영역
  - 제목 입력
  - 카테고리 선택
  - 상태 선택

**핵심 액션**
- 목표 생성
- 목표 상태 변경
- 진행률 수정
- 상위/하위 목표 연결
- 목표를 일정 후보로 전환

**연결 API**
- 현재 확정
  - `GET /api/goals`
  - `POST /api/goals`
- 이후 확장 권장
  - `PUT /api/goals/:id`
  - `DELETE /api/goals/:id`

**상태 설계**
- 기본 목표 seed가 있더라도 사용자가 직접 편집 가능한 구조
- 목표가 많아지면 접기/펼치기와 필터가 필요
- 빈 상태에서는 첫 목표 생성 가이드를 보여준다

**디자인 방향**
- 파일 탐색기형 트리와 상세 패널의 2단 레이아웃
- 대시보드보다 더 밀도 높게 정보를 배치해도 된다
- 진행률은 검정/회색 기본, 위험 상태만 강조색 사용
- 목표와 일정의 연결 관계를 `chip` 또는 `linked block` 형태로 드러낸다

**모바일 규칙**
- 트리와 상세 패널을 탭 전환 방식으로 분리
- 새 목표 생성은 하단 고정 CTA 또는 모달로 처리

### 6.4 `/schedule` 일정 페이지

**페이지 목적**
- 주간 시간표를 더 자세히 검토하고, import 결과를 확인하며, 필요 시 수동 보정하는 화면이다.
- 대시보드의 요약 시간표보다 더 깊은 일정 편집을 담당한다.

**핵심 사용자 질문**
- 이번 주 블록 배치가 실제 생활 리듬과 맞는가
- import 결과가 잘못 분류된 곳은 없는가
- 어떤 시간대가 비어 있고 어떤 시간대가 과밀한가

**반드시 들어가야 할 영역**
- 상단 주간 상태 바
  - 총 블록 수
  - 카테고리별 분포
  - 이번 주 수정 시각
- 주간 시간표 본문
  - 요일별 컬럼
  - 시간 블록
  - 카테고리 표시
  - note 확인
- import 검수 패널
  - 원본 텍스트
  - Gemma 변환 결과
  - 반영 전 미리보기
- 수동 편집 패널
  - 활동명
  - 시작/종료 시간
  - 카테고리
  - 메모

**핵심 액션**
- 주간 일정 새로고침
- 텍스트 import
- 기존 일정 덮어쓰기 여부 선택
- 단일 블록 수정
- note 추가

**연결 API**
- 현재 확정
  - `GET /api/schedule/week`
  - `POST /api/schedule/import`
- 이후 확장 권장
  - `PUT /api/schedule/block/:id`
  - `DELETE /api/schedule/block/:id`

**상태 설계**
- import 중에는 버튼을 잠그고 진행 상태를 명확히 표시
- 변환 실패 시 `gemma4` 문제와 API 실패를 구분해 보여준다
- 빈 주간 일정에서는 seed된 기본 루틴 또는 직접 생성 유도 둘 중 하나를 보여준다

**디자인 방향**
- 대시보드보다 더 많은 편집 컨트롤을 허용한다
- 왼쪽은 결과, 오른쪽은 입력과 검수 흐름으로 나눈다
- 하루 전체를 한눈에 보는 안정감이 중요하므로 카드보다 컬럼 일관성을 우선한다
- 카테고리 색은 과하지 않게 유지하되 `WORK`, `GROWTH`, `SLEEP`, `ADMIN`은 빠르게 구분돼야 한다

**모바일 규칙**
- 요일별 스와이프 또는 아코디언 구조
- import 패널은 하단 시트로 분리

### 6.5 `/settings` 설정 페이지

**페이지 목적**
- 앱의 정책과 연결 상태를 관리한다.
- 현재 상태를 설명하기보다, 변경 가능한 옵션을 안전하게 제어하게 한다.

**핵심 사용자 질문**
- Google 연결이 가능한가
- 조용한 시간과 개입 기준은 어떻게 돼 있는가
- 사용 중인 로컬 동작 조건은 무엇인가

**반드시 들어가야 할 영역**
- 연결 상태 카드
  - Google OAuth 상태
  - 콜백 URL
  - 마지막 동기화 시각
- 일정 정책 카드
  - 조용한 시간 시작/종료
  - 버퍼 분
  - 초과 근무 기준
  - 빈 시간 개입 기준
- 운영 정보 카드
  - 현재 백엔드 주소
  - 현재 사용 모델
  - API 문서 바로가기

**핵심 액션**
- Google 연결 재시도
- 설정 저장
- 기본값 복원

**연결 API**
- `GET /api/auth/session`
- `GET /api/auth/google/start`
- `GET /api/settings`
- `PUT /api/settings`

**상태 설계**
- 저장 중 상태와 저장 완료 토스트를 분리
- Google 미설정 상태를 실패처럼 보이게 하지 않는다
- 잘못된 시간 형식, 범위 오류는 인라인 검증으로 처리한다

**디자인 방향**
- 한 화면에 너무 많은 폼을 두지 말고 카드 단위로 나눈다
- 설정값은 설명보다 입력 필드가 먼저 보이게 한다
- 중요하지만 자주 안 바꾸는 값은 접기 섹션으로 내린다

**모바일 규칙**
- 카드 세로 스택
- 저장 버튼은 하단 고정 영역 검토

### 6.6 `/onboarding` 온보딩 페이지

**페이지 목적**
- 첫 사용자를 “API는 살아 있는데 아무것도 안 보이는 상태”에서 “즉시 써볼 수 있는 상태”까지 데려간다.

**핵심 사용자 질문**
- 무엇부터 연결해야 하는가
- Google 없이도 시작 가능한가
- 기본 루틴과 목표는 어떻게 넣는가

**반드시 들어가야 할 단계**
- 1단계: 제품 작동 방식 요약
- 2단계: Google 연결 또는 건너뛰기
- 3단계: 기본 루틴 또는 일정 텍스트 입력
- 4단계: 첫 목표 생성
- 5단계: 설정 기본값 확인
- 6단계: 대시보드 진입

**핵심 액션**
- Google 연결
- 건너뛰기
- 기본 루틴 입력
- 첫 목표 저장
- 대시보드 시작

**연결 API**
- `GET /api/onboarding/status`
- `GET /api/auth/google/start`
- `GET /api/settings`
- `PUT /api/settings`
- `GET /api/goals`
- `POST /api/goals`
- `GET /api/schedule/week`
- `POST /api/schedule/import`

**상태 설계**
- 단계 완료 여부를 상단 진행 표시로 드러낸다
- Google을 건너뛰어도 실패처럼 보이지 않아야 한다
- 어느 단계에서 나가도 이후 다시 이어서 시작할 수 있어야 한다

**디자인 방향**
- 대시보드보다 친절해야 하지만 랜딩처럼 길어지면 안 된다
- 한 번에 하나의 단계에 집중하게 한다
- 설명 카드와 입력 카드를 나눠 가독성을 높인다

**모바일 규칙**
- 완전한 단계형 플로우
- 하단 고정 CTA 사용 가능

### 6.7 `/api-docs` API 문서 페이지

**페이지 목적**
- 프론트 개발자와 운영자가 현재 실제로 쓰는 API 계약을 빠르게 확인하게 한다.
- Swagger 대체가 아니라 “현재 화면이 어떤 계약을 쓰는지”를 보여주는 내부 문서 페이지다.

**핵심 사용자 질문**
- 이 화면은 어떤 엔드포인트를 쓰는가
- 요청/응답 형식은 무엇인가
- 오류 응답은 어떤 shape인가

**반드시 들어가야 할 영역**
- 상단 요약
  - Base URL
  - 인증 방식
  - 로컬 모델 정보
- 기능 그룹 섹션
  - Auth & Session
  - Onboarding
  - Goals
  - Schedule
  - Settings
- 공통 에러 응답 섹션

**핵심 액션**
- 대시보드로 복귀
- 관련 페이지로 이동

**연결 데이터**
- `frontend/shared/content/apiReference.ts`
- `docs/api_reference.md`

**디자인 방향**
- 운영 화면보다 문서성이 더 강해야 한다
- 하지만 여전히 제품 내부 페이지처럼 보여야 한다
- 코드블록과 태그는 명확하게 보이되 화면이 개발자 전용 콘솔처럼 어두워질 필요는 없다

**모바일 규칙**
- 기능 그룹을 세로 아코디언으로 접기
- 긴 응답 shape는 수평 스크롤 또는 줄바꿈 허용

### 6.8 `/auth/callback` 인증 콜백 페이지

**페이지 목적**
- Google 인증 이후 세션 상태를 확인하고 사용자를 메인 화면으로 복귀시킨다.
- 사용자가 오래 머무르는 페이지가 아니라 짧은 전이 화면이다.

**핵심 사용자 질문**
- 로그인이 제대로 됐는가
- 지금 어디로 이동하는가

**반드시 들어가야 할 영역**
- 로딩 인디케이터
- 현재 상태 문구
- 실패 시 재시도 안내 또는 대시보드 복귀 버튼

**핵심 액션**
- 자동 세션 확인
- 자동 대시보드 이동
- 실패 시 수동 복귀

**연결 API**
- `GET /api/auth/session`

**디자인 방향**
- 가장 단순해야 한다
- 어두운 별도 테마를 쓰지 않는다
- 중앙 카드 1장과 상태 문구만으로 끝낸다

**개선 필요 사항**
- 현재 구현은 이전 테마 잔재가 있다
- 대시보드와 같은 밝은 토큰으로 통일해야 한다

### 6.9 `/mockups/today-workspace` 목업 페이지

**페이지 목적**
- 운영 중인 실제 페이지가 아니라, 더 공격적인 상호작용과 개입형 UX를 실험하는 내부 참고 화면이다.

**반드시 들어가야 할 것**
- 현재 실험 중인 고밀도 워크스페이스 개념
- 포커스 세션
- 개입 카드
- 오늘 타임라인
- 위험 목표 요약

**사용 원칙**
- 일반 사용자 진입 경로에 노출하지 않는다
- 실제 제품 페이지 설계 검토용 레퍼런스로만 쓴다

**디자인 방향**
- 실험적일 수 있으나, 운영 페이지에 채택할 수 있는 패턴만 추출한다
- 완성 제품보다 강한 대비와 고밀도 레이아웃을 허용한다

## 9. 페이지별 성공 기준

### 9.1 대시보드
- 사용자가 5초 안에 일정, 연결 상태, 목표 상태를 파악할 수 있다.
- import와 목표 추가가 별도 설명 없이 가능하다.

### 9.2 설정
- Google 연결 가능 여부와 설정 저장 여부를 한 번에 이해할 수 있다.

### 9.3 온보딩
- Google 없이도 대시보드 첫 진입까지 막히지 않는다.

### 9.4 목표
- 사용자가 트리 구조를 잃지 않고 하위 목표를 추가할 수 있다.

### 9.5 일정
- import 결과를 검수하고 필요한 수정 지점을 빠르게 찾을 수 있다.

### 9.6 API 문서
- 프론트 개발자가 코드 대신 문서만 보고 요청/응답 shape를 파악할 수 있다.

## 10. 정리

- 루트는 페이지가 아니라 진입점이다.
- 메인 제품은 `/dashboard`다.
- 운영형 페이지와 문서형 페이지를 분리하되 같은 디자인 체계를 쓴다.
- 이후 확장 페이지는 `온보딩 -> 설정 -> 목표 -> 일정` 순으로 늘리는 것이 맞다.
- 다시 랜딩 중심 구조로 돌아가면 제품 성격이 흐려진다.
